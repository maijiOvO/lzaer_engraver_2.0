# dev_tools/outputs — 开发测试输出目录规范

> **版本:** 2.0 | **日期:** 2026-07-01
>
> 本文档定义所有开发过程中产生的测试输出文件（图像、中间数据、报告）的组织规则。
> 所有测试脚本必须遵守此规范输出结果。

---

## 一、设计原则

**算法版本优先于输入图像。** 开发者在不断迭代算法，每一次运行通常涉及多张图片，核心诉求是对比同一算法在不同版本/参数组合下各图片的效果差异。因此目录结构以"版本"为第一级，以"城市（输入图像）"为第二级。

## 三、SAM 版本目录命名规范

### 3.1 版本目录名

```
格式：n{层数}_f{frame百分比}_i{island阈值}_{方法}
```

| 参数 | 含义 | 示例值 |
|------|------|--------|
| `n` | 分割层数 | 3, 4, 5 |
| `f` | frame 宽度百分比 | 50 |
| `i` | 最小岛屿面积阈值（px²） | 100 |
| `{方法}` | 后处理方法 | `std`(标准), `gf`(导向滤波边缘吸附) |

**特殊版本目录：**

| 目录名 | 含义 |
|--------|------|
| `v1_default` | 早期默认算法（无明确参数标记） |
| `v2_improved` | 第二代改进算法 |
| `depth_only` | 仅有深度数据，未跑 SAM 分割 |
| `exp_{实验名}` | 实验性输出 |
| `_archive/` | 已废弃的旧版本归档 |

### 3.2 每个城市子目录内部结构

```
{n_...}_{method}/{城市}/
├── depth/                                   # 深度数据
│   ├── {城市}_depth.npy                     # 原始深度 NumPy 数组
│   ├── {城市}_depth.png                     # 深度热力图
│   ├── {城市}_depth_raw.png                 # 原始深度灰度图
│   ├── {城市}_depth_gradient.png            # 深度梯度图
│   └── {城市}_depth_with_boundaries.png     # 带边界的深度图
├── {城市}_{参数组合}_frame_0..{n-1}.png     # 各层外框线预览
├── {城市}_{参数组合}_mask_0..{n-1}.png      # 各层 mask 预览
├── {城市}_{参数组合}.png                     # 叠加预览图
├── {城市}_sam_region.npz                    # SAM 分割区域数据
└── {城市}_{参数组合}_manifest.json          # 运行元信息（可选）
```

### 3.3 文件命名

```
格式：{城市}_n{层数}_f{frame百分比}_i{island阈值}_{方法}_{产物}.{扩展名}

产物类型：
  frame_{i}    第 i 层外框线预览（去掉了外框的纯内容）
  mask_{i}     第 i 层 mask 预览（含外框）
  (无后缀)      彩色叠加图
  depth.{npy|png}  深度数据
```

## 四、单步骤输出规范（canny / denoise / connectivity / svg）

这四个目录结构简单，每城市对应一个或少数几个文件，无需按版本分层。

### 4.1 命名格式

```
{城市}_{步骤标记}.{扩展名}
```

| 步骤 | 标记 | 示例 |
|------|------|------|
| canny | `canny` | `上海_canny.png` |
| denoise | `denoised_a{area}` | `上海_denoised_a4.png` |
| connectivity | `connected` | `上海_connected.png` |
| svg | `.svg` | `上海.svg` |

## 五、测试运行标准

### 5.1 test_runs（自动化全流程测试）

```
test_runs/{YYYY-MM-DD_HHMMSS}/
├── test_report.md           # 测试报告（必须包含：测试时间、参数、结果摘要）
├── image_id.txt             # 测试图像标识
├── api_outputs/             # API 原始响应输出
├── api_results/             # API 处理结果
└── dev_tools_outputs/       # 开发工具直接输出
```

### 5.2 integration_tests（手动集成测试）

```
integration_tests/{场景名称}/
├── 1_原图.jpg              # 步骤1：原始输入
├── 2_叠加图.png            # 步骤2：处理叠加
├── 3_图层{序号}_{位置}.png  # 步骤3：分层结果
├── 4_深度热力图.png         # 步骤4：深度可视化
└── ...
```

## 六、归档规则

### 6.1 何时归档

以下情况应将旧结果移至 `_archive/`：

- 算法核心逻辑发生重大变更（如从 GrabCut 切换到导向滤波）
- 参数组合被新方案完全取代
- 废弃的实验性输出不再有参考价值

### 6.2 归档格式

```
sam/_archive/{版本目录名}/{城市}/
```

### 6.3 归档记录

每次归档必须在 commit message 中注明原因。

## 七、禁止事项

1. **禁止将输出文件直接放在 sam/ 或其他顶层** — 必须归入对应的管线/版本子目录
2. **禁止使用中文目录名** — 除城市名称外，版本/参数目录全为英文
3. **禁止混淆参数版本** — 不同参数组合的结果必须分目录存放
4. **禁止在脚本中硬编码输出路径为绝对路径** — 使用相对于 `dev_tools/outputs/` 的路径
5. **禁止覆盖已有结果** — 新参数跑新目录，避免覆盖历史数据
6. **禁止城市优先分类** — 不要按"城市 → 版本"反向组织；版本必须在第一级
